Project 03

대규모 설비 이벤트 처리 시스템 : SFEP

제조 공장 설비에서 발생하는 대규모 센서 이벤트를 Kafka 기반 비동기 구조로 처리하고, PostgreSQL 저장, 위험 이벤트 실시간 알림, Prometheus/Grafana 모니터링까지 구성한 이벤트 처리 시스템입니다.

GitHub Notion 목록으로

개요

SFEP(Smart Factory Event Processing System)은 제조 공장 내 설비에서 발생하는 온도, 진동, 전류, RPM, 전력 사용량 등의 센서 이벤트를 수집하고 분석, 저장, 알림까지 처리하는 프로젝트입니다.

실제 제조 현장에서는 수백에서 수천 대의 설비가 지속적으로 데이터를 생성하기 때문에 단순 저장보다 이상 징후를 운영자가 빠르게 확인하고 대응할 수 있는 실시간 모니터링 구조가 중요하다고 판단했습니다.

Direct DB Write, Kafka 도입, Consumer 병렬화, JDBC Batch Insert, Alert Consumer 분리, 모니터링 구성까지 단계별로 구조를 확장하며 병목을 찾고 성능을 개선하는 데 초점을 두었습니다.

프로젝트 시연

설비 이벤트 시뮬레이터

  • 설비 수, 설비당 이벤트 수, 장애 비율 사용자가 직접 원하는 수량의 이벤트 생성
  • 지속 부하 테스트 기능을 통해서, 수량이 아닌 시간을 기준으로 이벤트 생성 기능 구현

위험 이벤트 실시간 알림 UI

  • 이벤트 발생 시각, 알림 발행 시각, UI 표시 시각을 각각 기록
  • 서버 알림 지연 시간, UI 큐 지연 시간, 전체 표시 지연 시간을 분리 측정
  • 평균 지연 시간뿐 아니라 p95, p99 지연 시간까지 확인 가능하도록 구성

모니터링 지표

  • Server UP
  • HTTP Requests/sec
  • HTTP Average Response Time
  • JVM Memory Used
  • Hikari DB Connection
  • Process CPU Usage

역할

  • Spring Boot 기반 백엔드 아키텍처 설계
  • 가상 설비 이벤트 생성 로직과 Swing 기반 시뮬레이션 UI 구현
  • Direct DB Write 저장 구조 구현 및 한계 측정
  • Kafka Producer/Consumer 기반 비동기 이벤트 처리 구조 설계
  • Consumer Group 분리를 통한 저장 처리와 알림 처리 분리
  • JDBC Batch Insert 기반 대량 저장 성능 개선
  • Hikari Connection Pool, Consumer Concurrency, Kafka Poll 설정 튜닝
  • 위험 이벤트 판단 로직과 실시간 알림 UI 구현
  • Prometheus / Grafana 기반 모니터링 환경 구성
  • 단계별 성능 측정 및 병목 분석 문서화

개선 사항

Direct DB Write 구조의 대량 이벤트 저장 한계

상황 1
AS-IS

초기 구조에서는 시뮬레이터가 생성한 이벤트를 Spring 서버가 PostgreSQL에 직접 저장했습니다. 구현은 단순했지만 대량 이벤트가 한 번에 들어오면 요청 처리와 DB 저장이 강하게 결합되어 병목이 발생했습니다.

TO-BE

JDBC Batch Insert를 적용해 여러 이벤트를 묶어서 DB에 전달하도록 개선했습니다. JPA 엔티티 상태 관리 비용과 DB 왕복 횟수를 줄여 대량 Insert 처리량을 높였습니다.

방식처리 시간처리율
Direct DB Write53,611ms933 events/sec
JDBC Batch Insert1,849ms27,056 events/sec

이벤트 수집과 저장이 결합된 구조

상황 2
AS-IS

Direct DB Write 방식에서는 DB 저장이 느려질 경우 이벤트 수집 요청 자체도 함께 지연되는 구조였습니다. 순간적으로 이벤트가 몰리는 상황에서 저장 처리 흐름이 전체 수집 흐름을 막을 수 있었습니다.

  • DB 저장 지연이 요청 처리 지연으로 전파
  • 저장 작업 실패 시 전체 요청 흐름 실패
  • 수집 처리와 저장 처리를 독립적으로 확장하기 어려움
  • 순간적인 이벤트 폭주를 버퍼링하기 어려움
TO-BE

서버는 이벤트를 Kafka Topic에 먼저 발행하고, Consumer가 메시지를 비동기적으로 소비해 DB 저장을 담당하도록 분리했습니다. Kafka가 중간 버퍼 역할을 하면서 수집 흐름과 저장 흐름을 안정적으로 분리할 수 있었습니다.

단일 Consumer 처리 구조의 처리량 한계

상황 3
AS-IS

Kafka를 도입하더라도 Consumer가 단일 처리 구조이면 Topic에 쌓인 이벤트를 빠르게 처리하기 어렵습니다. 이벤트 생산 속도가 소비 속도보다 빨라질 경우 Consumer Lag이 증가하고 실시간 처리성이 떨어질 수 있었습니다.

TO-BE

Kafka Topic Partition을 6개로 두고 Consumer Concurrency를 3으로 조정해 병렬 처리 구조를 적용했습니다. 이후 max.poll.records, Batch Size, Hikari Connection Pool 설정을 조합해 대량 이벤트 처리 실험을 반복했습니다.

Kafka Consumer 병렬화 후 최대 50,000 events/sec 수준의 처리량을 확인했습니다.

저장 처리와 위험 알림 처리가 같은 흐름에 묶이는 문제

상황 4
AS-IS

위험 이벤트 알림이 DB 저장 흐름에 함께 묶이면 DB 저장이 느려지는 순간 알림 전달도 함께 지연될 수 있습니다. 설비 과열, 진동 이상, 전류 급증 같은 이벤트는 저장보다 운영자가 빠르게 인지하는 것이 더 중요했습니다.

TO-BE

Kafka Consumer Group을 저장 Consumer와 알림 Consumer로 분리했습니다. 같은 센서 이벤트를 각각 독립적으로 소비하도록 구성하고, 알림 전달은 AlertEventBus와 별도 Thread Pool을 통해 비동기로 처리했습니다.

항목p95 알림 지연
저장 흐름 중심 처리6,959ms
Alert Consumer 분리 후1,926ms

위험 알림 지연 시간을 정확히 측정하기 어려운 문제

상황 5
AS-IS

초기에는 이벤트 저장 처리 시간과 전체 처리율은 확인할 수 있었지만, 위험 이벤트가 발생한 뒤 운영자 화면에 표시되기까지의 알림 지연 시간을 정확히 확인하기 어려웠습니다.

TO-BE

알림 이벤트에 시간 정보를 추가하고 Swing 알림 UI에서 서버 알림 지연, UI 큐 지연, 전체 표시 지연을 구간별로 측정하도록 개선했습니다. 이를 통해 지연 원인이 위험 판단인지, UI 큐 적체인지, 전체 처리 흐름인지 구분할 수 있게 했습니다.

측정 항목계산식
서버 알림 지연alertPublishedAt - occurredAt
UI 큐 지연displayedAt - alertPublishedAt
전체 표시 지연displayedAt - occurredAt

성능 개선 실험을 검증할 운영 지표 부족

상황 6
AS-IS

초기 성능 측정은 Swing UI의 처리 시간과 서버 로그 중심이었습니다. 전체 처리 시간과 초당 처리율은 확인할 수 있었지만 처리 중 서버 내부의 병목 자원을 파악하기 어려웠습니다.

TO-BE

Spring Actuator, Prometheus, Grafana를 도입해 JVM, HTTP 요청, DB Connection, CPU 사용량 등의 메트릭을 수집하고 시각화했습니다. 처리량뿐 아니라 서버 자원 사용량을 함께 확인해 성능 개선 결과를 더 신뢰도 있게 검증했습니다.

성능 개선 과정이 단발성 측정에 그치는 문제

상황 7
AS-IS

초기에는 특정 이벤트 수를 한 번 전송하고 처리 시간을 확인하는 방식으로 성능을 측정했습니다. 이 방식은 단일 요청 기준 처리 성능은 확인할 수 있지만 실제 운영 환경처럼 이벤트가 지속적으로 발생하는 상황을 검증하기에는 부족했습니다.

TO-BE

설비 수, 설비당 초당 이벤트 수, 실행 시간을 설정해 일정 시간 동안 이벤트를 지속적으로 발생시키는 부하 테스트 기능을 추가했습니다. 이를 통해 지속 부하 상황에서 이벤트 처리량, 알림 지연 시간, 서버 자원 사용량을 함께 관찰했습니다.

예: 설비 수 100대, 설비당 초당 이벤트 수 50건, 실행 시간 1분 조건으로 지속 부하를 검증했습니다.

최종 핵심 성과

  • JDBC Batch Insert 적용으로 100,000건 처리 시간 53,611ms에서 1,849ms로 개선
  • 저장 처리율 933 events/sec에서 27,056 events/sec로 개선
  • Kafka Consumer 병렬화 후 최대 50,000 events/sec 처리 확인
  • 저장 Consumer와 Alert Consumer 분리로 p95 알림 지연 6,959ms에서 1,926ms로 개선
  • Prometheus/Grafana 기반 모니터링으로 성능 개선 실험의 병목 분석 가능

결과 및 소감

이 프로젝트를 통해 단순히 기능을 구현하는 것보다 병목을 수치로 확인하고 구조적으로 개선하는 과정이 중요하다는 점을 배웠습니다. 초기 Direct DB Write 구조는 구현은 단순했지만 100,000건 이벤트 처리 시 약 53초가 소요되어 대규모 이벤트 처리에는 적합하지 않았습니다.

이후 Kafka를 도입해 수집과 저장을 분리하고, Consumer 병렬화와 JDBC Batch Insert를 적용하면서 처리량을 단계적으로 개선할 수 있었습니다. 또한 저장 처리율만 보는 것이 아니라 위험 이벤트가 운영자에게 도착하기까지의 알림 지연 시간을 측정하면서, 실시간 시스템에서는 저장 성능과 알림 실시간성을 분리해서 바라봐야 한다는 점을 체감했습니다.

마지막으로 Prometheus와 Grafana를 통해 서버 내부 지표를 함께 확인하면서, 성능 개선은 단순히 빠르게 만드는 것이 아니라 어떤 병목을 어떤 지표로 확인했고 어떤 구조로 개선했는지 설명할 수 있어야 한다는 점을 배웠습니다.
